# Advanced Concept Design

Concept granularity, row structure, keyword strategies, and design patterns.

## Concept Granularity

A concept is a searchable unit in the KG. The granularity decision affects search quality and assembly performance.

### Three-Level Hierarchy

| Level | Granularity | What It Captures | Search Use |
|-------|------------|-----------------|------------|
| Database | Coarse | Purpose, origin, sub-domains, tool ecosystem | "What databases do we have for X?" |
| Schema | Medium | Domain, data tier, table groupings | "What commerce data do we have?" |
| Table | Fine | Columns, types, grain, relationships, metrics | SQL generation context |

### Concept-Per-Table (Default)

Each table gets one concept row. Simple, direct mapping.

**Pros:** Easy to maintain, 1:1 with source objects, clear provenance
**Cons:** Table-centric (some questions are concept-centric)

### Concept-Per-Business-Entity (Advanced)

A single concept spans multiple tables representing one business entity.

Example: "Customer" concept spans:
- `CRM.CUSTOMERS` (core attributes)
- `BILLING.CUSTOMER_SUBSCRIPTIONS` (subscription status)
- `SUPPORT.CUSTOMER_TICKETS` (support history)

**Pros:** Answers entity-level questions directly, better search relevance
**Cons:** Complex to maintain, requires cross-table understanding, harder to auto-generate

### Hybrid Approach (Recommended)

1. **Table-level concepts** for all tables (auto-generated by CRAWL + ENRICH)
2. **Entity-level concepts** for important business entities (manually defined or AI-synthesized)
3. **Schema/DB-level concepts** for exploration and routing

The `concept_level` attribute enables filtering at search time.

## Concept Row Design

### search_content Field

The most critical field — this is what CSS indexes and searches against.

**Formula:**
```
search_content = concept_name + " " +
                 description + " " +
                 keywords (space-separated) + " " +
                 column_names (space-separated) + " " +
                 column_comments (space-separated) + " " +
                 table_comment + " " +
                 sample_values (for enum columns)
```

**Quality tips:**
- Include synonyms in keywords (e.g., "revenue, income, sales, earnings")
- Include both technical and business names (e.g., "GL_ENTRIES, general ledger, journal entries")
- Keep search_content under ~2000 tokens for CSS efficiency

### keywords Array

Explicit search terms beyond what's in descriptions and column names.

**Sources:**
- Business glossary terms
- Acronyms and their expansions (e.g., "MRR, monthly recurring revenue")
- Common question patterns (e.g., "how many customers", "total revenue")
- Domain-specific jargon

**Generation strategies:**
- Tier 0: Extract from existing comments and column names
- Tier 2: AI_EXTRACT with prompt: "Generate 5-10 search keywords for this table"
- Tier 3: AI_COMPLETE with prompt: "What business questions could this table answer?"

### tables_yaml Field

Structured payload for Cortex Analyst context. Must include:

```yaml
- name: DB.SCHEMA.TABLE
  description: Human-readable purpose
  columns:
    - name: COLUMN_NAME
      data_type: VARCHAR/NUMBER/DATE/etc
      role: key|dimension|metric|timestamp|flag|text
      description: What this column represents
      sample_values: ["val1", "val2"]     # for enum columns
      is_enum: true|false
```

**Column roles:**
- `key`: Primary key, foreign key, surrogate key
- `dimension`: Categorical, groupable (region, status, category)
- `metric`: Numeric, aggregatable (amount, count, rate)
- `timestamp`: Time-based, filterable (created_at, order_date)
- `flag`: Boolean (is_active, has_subscription)
- `text`: Free-form text (description, notes)

### metrics_yaml Field

Pre-defined aggregations for common questions:

```yaml
- name: total_revenue
  expression: SUM(AMOUNT)
  description: Total revenue amount
  table: FINANCE_DB.CORE.INVOICES
- name: customer_count
  expression: COUNT(DISTINCT CUSTOMER_ID)
  description: Number of unique customers
  table: CRM.CORE.CUSTOMERS
- name: avg_order_value
  expression: AVG(ORDER_TOTAL)
  description: Average value per order
  table: SALES.CORE.ORDERS
```

## Concept Quality Checklist

Before indexing a concept:

- [ ] `concept_name` is human-readable (not just the table name)
- [ ] `description` is > 20 characters and describes purpose
- [ ] `keywords` has >= 3 relevant terms
- [ ] `search_content` includes all searchable text
- [ ] `tables_yaml` has column names, types, and roles
- [ ] `join_keys_yaml` populated if relationships exist
- [ ] `is_enum` set correctly for categorical columns
- [ ] `sample_values` populated for enum columns
- [ ] `enrichment_quality_score` >= 0.5

## Anti-Patterns

- **Too many concepts per table** — Don't create separate concepts for every column
- **Duplicate search content** — If two concepts have identical search_content, they'll compete in results
- **Missing join keys** — Without relationship info, assembly can't connect tables
- **Overly technical descriptions** — "Fact table with SCD2 dimensions" is useless for business users; prefer "Daily sales transactions with customer and product details"
- **No sample values** — For columns like STATUS, CATEGORY, REGION — sample values dramatically improve literal matching
